# System Contracts

These are the contracts that make the project portable. Preserve them unless you deliberately redesign the whole system.

## Display contract

The TV displays a single JPEG generated by the active renderer:

```text
/config/www/current.jpg
```

Home Assistant exposes it as:

```text
http://<home-assistant-host>:8123/local/current.jpg?t=<timestamp>
```

Rules:

- Always use a fresh query string when casting.
- Write temp files on the same filesystem as `current.jpg`, then atomically replace.
- When Chromecast ignores a refreshed JPEG, call `media_stop`, wait briefly, then `play_media` again.

## Renderer ownership

Only one renderer should actively own `current.jpg`.

| State | Active process | Output |
|---|---|---|
| Photos | `slideshow_gen.py` | photo + clock + weather |
| Music | `musica_slideshow.py` | album art + track metadata + optional recipe |

State transitions are controlled by Home Assistant automations, not the web app.

## Music metadata contract

Home Assistant writes a pipe-delimited file:

```text
/config/.music_state.txt
```

Format:

```text
state|media_content_id|media_artist|media_title|entity_picture|media_album_name
```

Example:

```text
playing|yt:abc123|Massive Attack|Teardrop|/api/media_player_proxy/media_player.SALA?...|Mezzanine
```

Rules:

- Use a `command_line` sensor for templating. `shell_command` does not render Jinja2.
- Strip or replace single quotes in metadata before writing the shell command.
- Trigger a forced `homeassistant.update_entity` before starting the music renderer.
- Add a small delay before and after that forced update to avoid mixed old/new metadata.

## Recipe overlay contract

The active recipe is one JSON file:

```text
/config/music_recipe.json
```

Schema:

```json
{
  "enabled": true,
  "title": "RECIPE NAME",
  "lines": [
    "ingredient or short step",
    "another short line"
  ]
}
```

Rules:

- The renderer must behave normally when the file is missing.
- The renderer must ignore the overlay unless `enabled` is `true`.
- The renderer should read the file repeatedly, not only at process start.
- Lines should already be TV-optimized before writing the file.
- Deleting the file is the normal disarm operation.

See [`../examples/music_recipe.example.json`](../examples/music_recipe.example.json).

## Recipe arming contract

Arming a recipe means:

1. Make recipe JSON available at the web app endpoint:

```text
GET /api/pending_recipe
```

2. Home Assistant pulls that JSON:

```text
curl -sf http://<app-host>:8095/api/pending_recipe -o /config/music_recipe.json
```

3. Home Assistant turns on the recipe gate:

```text
input_boolean.recipe_armed = on
```

4. Home Assistant starts the recipe window:

```text
timer.recipe_window
```

The web app does not write directly to the Home Assistant filesystem and does not need SSH.

## Recipe disarm contract

Disarming means:

1. Delete `/config/music_recipe.json`.
2. Turn off `input_boolean.recipe_armed`.
3. Cancel `timer.recipe_window`.

Recommended automation:

- Disarm when the timer expires and the speaker is not playing.
- Disarm after the speaker has been idle/off/unavailable for a grace period.
- Never disarm while the speaker is currently `playing`.

## Web app contract

The app provides:

| Route | Purpose |
|---|---|
| `GET /` | list recipes |
| `GET /r/<id>` | show a recipe and playlist controls |
| `POST /cook` | arm recipe and start playlist job |
| `GET /job/<id>` | poll playlist job status |
| `GET /quick?rid=<id>&list=<playlist_id>` | arm recipe and redirect to an existing playlist |
| `GET /api/pending_recipe` | expose latest armed recipe JSON for Home Assistant |
| `GET /healthz` | simple liveness check |

The app can fail to create a playlist and still succeed at arming the recipe.

## AI worker contract

The AI worker is replaceable. It must accept:

- recipe title
- user vibe/instructions
- access to a music service tool or API

It must return:

- a playlist URL, preferably a YouTube Music playlist URL

The current app expects to extract a URL matching:

```text
music.youtube.com/playlist?list=<id>
```

If you use a different music provider, change only the worker call and URL extraction path first.

## Entity placeholders

Replace these placeholders in Home Assistant snippets:

| Placeholder | Meaning |
|---|---|
| `media_player.SALA` | music speaker / trigger player |
| `media_player.COZINHA` | TV Chromecast / display player |
| `HA_HOST` | Home Assistant hostname/IP reachable by Chromecast |
| `APP_HOST:8095` | web app host reachable by Home Assistant |

## Timing invariants

Keep these unless you have evidence they are unnecessary in your environment:

- Wait before reading music metadata after the speaker starts.
- Force-update the metadata writer sensor.
- Wait again before launching the music renderer.
- Push the TV once shortly after launch.
- Push the TV a second time later to avoid the previous-track frame.

The second push looks ugly in YAML, but it avoids a very real Chromecast timing bug.

## Security boundaries

- Do not commit tokens, auth JSON, or exact private hostnames.
- Do not put an SSH private key in the web app.
- Prefer a Home Assistant long-lived token scoped by practical containment.
- Mount recipes and YouTube Music auth read-only.
- Treat Docker socket access as privileged. If exposed, place the app behind authentication.
